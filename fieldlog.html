CASE 001

Bulk customer-data recovery across 12 distributed sites

Situation

A French automotive group's customer records were silently losing phone numbers at every dealership. The live incremental sync, the integration the whole pipeline depended on, had gone dark group-wide after a downstream change. The nightly batch was erroring and no record was being completed.

Approach

I isolated the failure to one variable at a time and proved the incremental endpoint was dead for any watermark. Rather than edit the broken stored configuration, I built an external reconciliation process in Python: extract the affected records from the CRM, fetch the missing values from the source system matched by exact record ID, and inject them back through the API, writing only currently empty fields.

Result

Recovered 1,830 missing records across the 12 sites with a 100% match rate on record ID, zero destructive writes, and the batch job back to 0 errors. The broken incremental path was left dead and replaced by a process I could verify end to end.

CASE 002

Real-time gateway stall on a streaming-only upstream

Situation

An internal app's AI features hung indefinitely. Requests to the model gateway never returned. The infrastructure looked healthy: no server errors, no timeouts. A classic "not a network problem" symptom that points at the wrong layer.

Approach

I traced the request path end to end and found the real cause: the client was requesting a non-streaming response from an upstream that only speaks Server-Sent Events. No sudo, no network change, no vendor ticket. The diagnostic discipline, prove the real cause before touching anything, is the whole point of the engagement.

Result

Streaming restored with a one-line client change. No infrastructure rebuild. The dependency was never broken. It was being asked the wrong question.
