CASE 001

Bulk customer-data recovery across 12 distributed sites

Situation

A French automotive group's customer records were silently losing phone numbers at every dealership. The live incremental sync, the integration the whole pipeline depended on, had gone dark group-wide after a vendor-side change. 12 sites, no recovery path, and the batch job was erroring on every run.

Approach

Isolated the failure to one variable at a time and proved the incremental endpoint was dead for any watermark. Built an external ETL in Python. Extract the affected records from the CRM, fetch the missing data from the DMS matched by exact record ID (no date-window guessing), then re-inject only the empty fields via the API. Conservative staged rollout. Fix one site, verify zero errors, then propagate.

Result

Recovered 1,830 missing records across the 12 sites with a 100% match rate on record ID, zero destructive writes, and the batch job back to 0 errors. The broken incremental path was left untouched. Worked around, not rebuilt.

CASE 002

Real-time gateway stall on a streaming-only upstream

Situation

An internal app's AI features hung indefinitely. Requests to the model gateway never returned. The infra looked healthy, no server errors, no timeouts. Classic not a network problem symptom.

Approach

Traced the request path end to end and found the real cause: the client was requesting a non-streaming response from an upstream that only speaks Server-Sent Events. No sudo, no network change needed. The fix was client-side: request SSE streaming from the gateway. Diagnosed the protocol mismatch instead of concluding the dependency was broken.

Result

Streaming restored with a one-line client change. No infrastructure rebuild, no vendor ticket. The diagnostic discipline, prove the real cause before touching anything, is the whole point of forward deployment.
